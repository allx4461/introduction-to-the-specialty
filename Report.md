# Технический отчет по проекту: Microservice ASCII Aquarium & Chat

---

## 1. Введение и цель проекта

Идеей данного проекта является создание интерактивного пространства для отдыха и ностальгии по «старому интернету». Это достигается за счет ASCII-графики, которая не нагружает интерфейс излишними деталями. Проект представляет собой симуляцию аквариума в реальном времени, совмещенную с анонимным чатом сгорающих сообщений, где пользователи могут делиться своими мыслями.

**Цель проекта:**
* Проектирование и развертывание распределенной микросервисной системы.
* Практическое освоение современных подходов к DevOps: контейнеризации, оркестрации, автоматического тестирования и статического анализа безопасности кода и зависимостей в рамках CI/CD.

---

## 2. Архитектура системы

Приложение построено на базе микросервисной архитектуры. Схема взаимодействия компонент:

```mermaid
graph TD
    User([Браузер пользователя]) -->|1. Запрос HTML/JS (порт 80)| Frontend[Контейнер Nginx: Frontend]
    User -->|2. Анимация аквариума (порт 8080)| BackendCPP[Контейнер C++: Bulbul]
    User -->|3. Чат и оправдания (порт 5000)| BackendPy[Контейнер Python: Informer]
    BackendPy -->|4. Внешний запрос| ExternalAPI[Внешнее API оправданий]

    subgraph Docker Compose Network
        Frontend
        BackendCPP
        BackendPy
    end
```

### микросервисы:
1. **Frontend (Nginx Alpine):**
   * Обслуживает статический контент (`index.html`, стили, JS) на порту `80`.
   * JavaScript в браузере динамически определяет хост (`window.location.hostname || 'localhost'`) и делает асинхронные запросы к API бэкендов.
2. **C++ Backend (httplib):**
   * Располагается в папке `bulbul/`, работает на порту `8080`.
   * Генерирует ASCII-кадры аквариума, обрабатывает логику движения сущностей (рыб и пузырьков), а также предоставляет HTTP API для добавления рыб (`/add_fish1` .. `/add_fish5`) и очистки аквариума (`/clear`).
3. **Python Backend (FastAPI):**
   * Располагается в папке `informer/`, работает на порту `5000`.
   * Содержит анонимный чат (запись `POST /messages`, чтение `GET /messages`) и проксирует запросы к стороннему API оправданий (`GET /excuse`).

## 3. Контейнеризация (Docker)

Для обеспечения воспроизводимости запуска и изоляции окружения все компоненты упакованы в Docker-контейнеры.

### Оптимизация образов и сборки:
* **Многоэтапная сборка (Multi-stage) C++:** 
  В [bulbul/Dockerfile](file:///bulbul/Dockerfile) сборка разделена на два этапа. На этапе `builder` ставится компилятор `g++`, `cmake` и `make` для сборки бинарного файла `aquarium_app`. На финальном этапе `runtime` используется чистый легковесный образ `ubuntu:24.04`, куда копируется только скомпилированный бинарник. Это позволяет уменьшить финальный образ и не тащить в продакшн инструменты компиляции.
* **Кэширование зависимостей Python:** 
  В [informer/Dockerfile](file:///informer/Dockerfile) сначала копируется файл `requirements.txt` и запускается `pip install`, а только затем копируется сам код приложения. Это позволяет Docker использовать кэш слоев: при изменении кода бэкенда зависимости не скачиваются заново.
* **Легковесный фронтенд:** 
  Используется образ `nginx:alpine` (размером менее 10 МБ), который просто раздает статику.
* **Исключение мусора (`.dockerignore`):**
  В каждом сервисе настроены файлы исключений. Они предотвращают попадание кэша компиляции, тестов и конфигураций IDE внутрь образов, минимизируя размер контекста сборки.

**Сборка и запуск оркестратора:**
```bash
docker compose up --build -d
```

## 4. Автоматизация тестирования

Для контроля качества кода реализовано два типа тестирования, запускаемых автоматически как локально, так и в CI/CD.

### 4.1. Модульные тесты (C++ Unit Tests)
Тестируют ключевую логику симулятора аквариума (класса `World`) независимо от сетевого HTTP-сервера. 
* **Файл теста:** [bulbul/tests/test_world.cpp](file:///bulbul/tests/test_world.cpp)
* **Покрытие:** границы аквариума (`inside`), инициализация размеров и очистка сущностей (`clear`).

*Пример кода теста:*
```cpp
World world(100, 50);
assert(world.inside(10, 10, 5, 5) == true);  // Внутри границ
assert(world.inside(-1, 10, 5, 5) == false); // Выход за левую границу
```
*Локальный запуск:*
```bash
cd bulbul
g++ -std=c++17 -Isrc src/world.cpp src/actor.cpp src/entity.cpp src/fishes.cpp src/canvas.cpp src/bubble.cpp tests/test_world.cpp -o run_tests -pthread
./run_tests
```

### 4.2. Интеграционные тесты API (Python / Pytest)
Проверяют работоспособность сетевых эндпоинтов Python бэкенда.
* **Файл теста:** [informer/test_main.py](file:///informer/test_main.py)
* **Покрытие:** HTTP эндпоинты `/excuse` и `/messages` (отправка `POST` и последующее считывание через `GET`).

*Локальный запуск:*
```bash
pytest informer/
```

### 4.3. Интеграция в CI (GitHub Actions)
В файле [.github/workflows/config.yml](file:///.github/workflows/config.yml) настроен автоматический пайплайн, запускающийся при каждом Pull Request в ветку `main`. Шаги пайплайна:
1. Проверка форматирования Python-кода линтером `flake8`.
2. Запуск интеграционных тестов `pytest`.
3. Проверка сборки продакшн C++ бэкенда.
4. Сборка и запуск Unit-тестов C++.

## 5. Безопасность и статический анализ (SAST/SCA)

Качество безопасности кода автоматизировано на этапе разработки при помощи сканеров уязвимостей в CI.

### 5.1. SAST (Анализ исходного кода через Semgrep)
Пайплайн использует инструмент **Semgrep** для автоматического выявления небезопасных конструкций кода.

* **Выявленная уязвимость:** Вызов `requests.get()` без указания таймаута (`timeout`).
* **Чем это опасно:** Если сторонний сервер API оправданий зависнет или начнет отвечать бесконечно медленно, поток Python-сервера заблокируется в ожидании ответа. Это может быстро парализовать работу всего чата (уязвимость типа Denial of Service).
* **Решение:** Добавлен лимит времени ожидания `timeout=5.0`.
  ```python
  # Было:
  response = requests.get("https://excuser-three.vercel.app/v1/excuse")
  # Стало (Безопасно):
  response = requests.get("https://excuser-three.vercel.app/v1/excuse", timeout=5.0)
  ```

### 5.2. SCA (Анализ сторонних зависимостей через Trivy)
Инструмент **Trivy** интегрирован в CI для сканирования сторонних библиотек на наличие известных уязвимостей (CVE).

* **Устранение рисков:** 
  Изначально зависимости в `requirements.txt` были не зафиксированы, что позволяло скачиваться уязвимым или несовместимым версиям библиотек. В рамках ДЗ 7 версии были строго ограничены безопасными релизами:
  ```text
  fastapi>=0.109.0
  uvicorn>=0.27.0
  requests>=2.31.0
  ```
  Это гарантирует, что в контейнер собираются только проверенные пакеты без известных критических CVE.

### 5.3. Защита от XSS на фронтенде
При отображении сообщений чата в браузере внедрена защита от Cross-Site Scripting (XSS). Текст сообщений выводится через свойство `.textContent` вместо `.innerHTML`:
```javascript
// Безопасное добавление текста сообщения на страницу
const textSpan = document.createElement('span');
textSpan.className = 'chat-text';
textSpan.textContent = msg.text; // Браузер отобразит HTML-теги как текст, а не выполнит их
```

## 6. Выводы и перспективы развития

В ходе выполнения проекта были освоены навыки создания распределенных контейнеризованных систем. Разработан отказоустойчивый микросервис с автоматическими проверками качества.

**Навыки, полученные в процессе разработки:**
* Настройка взаимодействия микросервисов в рамках единой сети Docker Compose.
* Реализация автоматических проверок (линтеры, тесты) в GitHub Actions.
* Работа со сканерами безопасности SAST и SCA.

**Направления для будущего развития проекта:**
1. **Персистентность данных:** Использование СУБД (например, PostgreSQL или Redis) для хранения сообщений чата вместо хранения в оперативной памяти (RAM) Python-процесса.
2. **Авторизация:** Внедрение JWT-токенов для идентификации пользователей в чате.
3. **Безопасность сети:** Настройка HTTPS через обратный прокси-сервер (Nginx) с SSL-сертификатами Let's Encrypt для шифрования трафика чата.
